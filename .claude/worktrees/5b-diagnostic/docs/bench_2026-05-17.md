# MLRift vs PyTorch — 2026-05-17 post-reboot benchmark snapshot

**Hardware:** AMD Radeon RX 7800 XT (gfx1100, 16 GiB VRAM, ~480 GB/s DRAM)
+ AMD Ryzen 9 7900X (24-thread) + 30 GiB host RAM.

**Stack:** PyTorch 2.9.1+rocm6.4 baselines from `docs/bench_2026-05-13.md`
on the same hardware. MLRift commit `5dc446b` (post slice 6.8d KFD wedge fix
+ 6.8e probe rig + 7.4 host RSS dealloc + Llama-3B bf16 default flip + slice
7.6a/b multi-accumulator + VOPD `v_dual_fmac_f32` packing in the phase-3
mega-kernel inner gemv).

**Workload:** 20-50 token greedy decode of `"hello"` (Qwen3 mks16 uses a
16-token bigram prompt for PLD-bench acceptance).

This is a *post-reboot re-verification* of every claim on the website's
benchmark tables — slice 6.8d means future processes no longer accumulate
KFD wedge state, so back-to-back benches are reproducible without reboot.

## Reproducer envs (post-reboot)

```bash
# Helper .co rebuild (always run after /tmp wipe or reboot)

MLRIFT_QWEN3_0_6B_DIR=/path/to/Qwen3-0.6B/model.safetensors \
MLRIFT_NATIVE_MEGAKERNEL=2 \
/tmp/qwen3_gen

# Qwen3-0.6B mks16+PLD peak (264 tok/s)
MLRIFT_QWEN3_0_6B_DIR=/path/to/Qwen3-0.6B/model.safetensors \
```

## Changes since `bench_2026-05-13.md`

1. **Llama-3.2-3B** added to the GPU scorecard at 40.7 tok/s, +28% over PT bf16
   GPU. Slice 8.4 (commit 455df3f) restored the mega-kernel from a 5.88 tok/s
   cputree regression; slice 7.4 (commit 447d220) saved 3.5 GiB host RSS. The
   Llama-3B bf16 mega-kernel path is now the **driver default** after commit
   f78e7c3 — no `MLRIFT_LLAMA3_3B_USE_BF16=1` env required.

2. **KFD wedge fix (slice 6.8d, commit a94b0d5)** prevents future processes
   from accumulating MES firmware wedge state. The chronic
   "hipDeviceSynchronize: timeout draining queue errno=30000000001" after
   back-to-back GPU runs is now structurally fixed.

3. **Mistral bf16 safetensors path** added in commit cd2dabd as an alternative
   to the Q8_0 GGUF path. Same tok/s, heavier host RSS (27.8 vs 14.0 GiB)
   because the safetensors are mmap'd AND cloned to heap. Used only for
   sub-ULP PT-divergence bisection.

4. **Mistral 22/23 vs PT** — fully bisected to reduction-tree topology
   (slice 6.8e probe rig). Kahan-compensated reduction in down_proj
   (`MLRIFT_DOWN_PROJ_KAHAN=1`) produces byte-identical output → proves the
   gap is *order of summation* not *precision of summation*. Same reduction
   family as Ollama/llama.cpp; structurally sub-ULP to chase PT exactly.

5. **Slice 7.6 — multi-accumulator + VOPD `v_dual_fmac_f32`** (commits
   252acb8 + 5dc446b). Phase-3 mega-kernel inner gemv was a 16-deep FMA
   chain through a single accumulator (slice 7.5 8× unroll). Split into 8
   independent accumulators with a 7-add final reduction, then packed
   adjacent FMA pairs into gfx11 VOPD `v_dual_fmac_f32` for RDNA3
   dual-issue. Tokens byte-identical to slice 7.5 on all 3 GPU models
   (PyTorch bit-exactness preserved). Phase-9 residual gemv (Mistral
   down_proj at K=14336) **retains slice-7.5 single-accumulator pattern**
   — multi-acc there wedges the KFD queue, root cause unknown,
   investigation pending. Measured (median of 3-4 runs each):
   * Mistral-7B Q8_0: 21.7 → **22.9** tok/s (+5.5%)
   * Qwen3-0.6B M=1 mega-kernel: 118 → **119.7** tok/s (+1.4%, within noise)
   * Llama-3.2-1B: 99.8 → 99.8 tok/s (flat — already at the bandwidth
     ceiling on its smaller K dims)

6. **Slice 7.7 — KFD wedge watchdog** (commit b5ffc20). 5-second
   time-based timeout in every dispatch + sync + wait path, replacing
   the old 30-billion-spin (≈30 s) timeout that fired into `kfd_die`.
   On wedge: one `[KFD-WEDGE]` diagnostic line, `_kfd_wedge_seen` flag
   set, all subsequent dispatch/sync calls no-op, `hipkfd_teardown`
   skips compute + boost queue `DESTROY_QUEUE` (which itself goes
   through MES and would also time out). Launcher exits in <100 ms vs
   the prior ~60 s opaque hang. Does NOT fix wedges (MES is still
   stuck; reboot required to clear) but turns the symptom from
   "everything is frozen for a minute" into "single clear error then
   immediate exit." Healthy-path tok/s unchanged.

7. **Slice 7.7b — launcher-exit gap closed** (commit eb0e73a). Found
   four post-wedge block sites slice 7.7 missed: `hipModuleLaunchKernel`
   back-pressure spin re-entrance, `hipkfd_teardown` boost-SDMA drain
   spin, `kfd_release_all_allocations` MES-routed UNMAP calls (the 115 s
   tail), and driver decode loops in 4 generate files. All gated under
   `_kfd_wedge_seen != 0`. Real-world validated 2026-05-17: launcher
   exited ~13 s after `[KFD-WEDGE]` (within 15 s acceptance threshold;
   the residual is a pre-existing SIGSEGV in cleanup — likely
   libamdhip64.so.7 atexit). Healthy-path bit-exactness preserved.

8. **Slice 7.9 — phase 9/17 structural split** (commit 3bcbb09).
   Definitive answer to the phase-9 multi-acc wedge: slice 7.8 F1
   (balanced tree + s_delay_alu) and F4 (skip multi-acc on down_proj
   only) both wedged identically — proving K=14336 is NOT the trigger;
   the wedge is the *shape* of `multi-acc dependent-VALU reduce →
   VMEM residual-load + add + store`. Phase 3 has the same multi-acc
   body but ends in wave-XOR + LDS + plain store and works fine.
   Slice 7.9 splits phase 9 (o_proj) and phase 17 (down_proj) into
   two pieces with a cross-WG barrier between them:
     gemv → intermediate scratch (plain store, no residual)
     → barrier
     → residual-add (streamed elementwise: load gemv + load residual + add + store)
   The gemv now matches phase 3's proven shape and gets the slice-7.6a
   multi-acc + slice-7.6b VOPD packing. The intermediate buffer reuses
   `b_attn_q_g` (kernarg 0x98, dead from phase 7 onward, size
   q_dim*4 = HIDDEN*4 = 16 KiB exact match) — zero extra allocation.
   * Mistral-7B Q8_0: 22.9 → **24.9** tok/s (+8.7%, 3-run median)
   * Llama-1B / Llama-3B / Qwen3: byte-identical .co files (split is
     Mistral-only via the `MEGA_HIDDEN==4096` gate; other archs keep
     the monolithic phase-9 emitter)
   Tokens still 22/23 bit-exact vs PyTorch canonical (the pos-6 ULP
   gap lives in lm_head reductions, unaffected by phase 9/17 split).
   Two extra cross-WG barriers per layer × 32 layers ≈ 0.8-1.6%
   overhead, dwarfed by the multi-acc unlock gain.

## Known caveats / open follow-ups

- **Qwen3-0.6B default invocation (no envs) still hits the CPU forward path
  at ~30 tok/s** — only the GPU build with `MLRIFT_NATIVE_MEGAKERNEL=2`
  (or the speck-K reproducer) activates the mega-kernel. Slice 8.5 (commit
  b69dc0b, 2026-05-17) collapsed the prior 3-env opt-in into a single env.

- **Qwen3 mks16 long-run wedge** at step ~14 on bigram prompts (after PLD
  acceptance drops to 1/16, the per-op fallback chain hammers the queue and
  trips a sync timeout intra-process). Different from the 6.8d MES wedge —
  needs separate investigation.

- **PT Qwen3-0.6B GPU baseline** not directly measured; uses Qwen3.5-0.8B's
  49.8 tok/s as a proxy for the comparison ratios. Would benefit from a
  direct measurement once a ROCm-enabled PT venv is set up again.

- **MLRift VRAM** numbers in the table are estimated from prior `mem_info_vram_used`
  /sys/class readings; today's run did not capture per-process VRAM directly.
  The KFD alloc summary print landed via slice 6.8d but is not surfaced via
  the per-process accounting yet.

- **Slice 7.6 phase-9 (residual gemv) wedge** — applying the multi-accumulator
  body to `_emit_phase9_gemv_residual_inline` wedges the KFD queue on
  Mistral's down_proj (k=14336, 28 inner iters × 16 rows × 256 WGs) at
  ≥4 accumulators. Llama-1B and Qwen3 don't surface the issue (smaller K).
  Bisection ruled out bank conflicts, VGPR budget, branch range, and
  VALU→LDS hazard — different from any known wedge mode. Phase-9 retained
  the slice-7.5 single-accumulator pattern, which means Mistral's biggest
  hot path (down_proj is ~50% of decode time) didn't get the VOPD gain.
