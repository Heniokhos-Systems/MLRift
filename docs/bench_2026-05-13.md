# MLRift vs PyTorch — 2026-05-13 benchmark snapshot

**Hardware:** AMD Radeon RX 7800 XT (gfx1100, 16 GiB VRAM, ~480 GB/s DRAM)
+ 24-thread x86_64 + 30 GiB RAM.

**Stack:** PyTorch 2.9.1+rocm6.4, transformers 5.8.0 (memory baselines from
2026-05-07 on same hardware). MLRift commit `24dd54f`.

**Workload:** 20-token greedy decode of `"hello"` (or PLD-bench bigram-rigged
prompt where noted).

M=1 still 28% below PT on real prompts — closes when inner-loop opts

on M=1 per the 5× floor-gap analysis ([project_megakernel_inner_loop_5x_gap.md]).

## What ships in destroy-PT today

- **GPU bit-exact validation:** 7/7 models (Llama-1B/3B + Mistral-7B +
  Gemma2-2B + Gemma3-1B + Qwen3-VL-4B + Qwen3.5-0.8B) match llama.cpp greedy
  on CPU. Llama-1B + Qwen3-0.6B additionally validated on GPU.
- **Single-stream beats PT on Qwen3-0.6B:** 216 vs ~50 = 4× (PLD-bench).
- **Single-stream beats PT on Llama-3.2-1B:** 64.5 vs 51.3 = +26% (PLD-bench).

## What's still PT-ahead

- Llama-1B real prompts (no PLD): 37 vs 51 = 0.72× — to be closed by inner-loop
  opt #1 (s_clause batching, projected M=1 → 125-150 tok/s)
- All non-Qwen3-0.6B / non-Llama-1B models on GPU: not yet ported. The destroy-PT
  pattern (per-layer mega-kernel via AST-walker, slice 4.19.x/5.x for Qwen3
  and Llama) needs to be parameterized per-arch for the remaining 5 models.

## Asymmetric advantages

- **VRAM:** every MLRift driver is CPU-only here. PT fp32 7B and PT fp32 4B both
  OOM on 16 GiB; MLRift runs them on host RAM.
- **RAM efficiency:** MLRift Mistral-7B uses 7.2 GiB RSS (Q8_0 stays
  quantized); PT bf16 needs 13.9 GiB VRAM (within 0.5 GiB of OOM).
- **No ROCm / HIP / hipcc in build chain** ([feedback_no_rocm_hip_hipcc]):
  every production .co comes from MLRift's `--emit-amdgpu-*-v2` AST-walker.
  hipcc-only `speck8`/`speck16` are scaffolded (slice 4.21/4.22, commit
  `24dd54f`) but not yet shipping a .co — asm buffer cap bump pending.
- **Bit-identity vs llama.cpp:** PT SDPA on ROCm has known small numerical
  drift; MLRift matches llama.cpp greedy exactly on every model.

## Methodology

- All MLRift CPU runs: `MLRIFT_LLAMA_N=20 timeout 300 /tmp/<driver>`
- MLRift GPU M=1: `MLRIFT_NATIVE_MEGAKERNEL=1 MLRIFT_LLAMA_N=20 /tmp/llama32_1b_gpu`
- MLRift GPU speck4+PLD: `MLRIFT_NATIVE_MEGAKERNEL=2 MLRIFT_PLD_BENCH=1 MLRIFT_LLAMA_N=20 /tmp/llama32_1b_gpu`
- PT GPU/CPU baselines: `/tmp/bench/bench_pt.py --model ... --n 20` (memory note 2026-05-07).
- tok/s for MLRift drivers = printed `tokens_per_sec_x1000 / 1000` (it's milli-tok/s).
